# Highload architectural patterns

## Асинхронность
- Используй `asyncio` для I/O-bound задач (сеть, БД, файлы)
- Синхронные блокирующие вызовы внутри async-кода оборачивай в `asyncio.to_thread()`
- Не создавай новый `event loop` вручную — используй `asyncio.run()` только на верхнем уровне

## Кэширование
- Кэшируй результаты дорогих вычислений и внешних запросов
- Применяй TTL — не храни кэш вечно
- Для in-process кэша используй `functools.lru_cache` / `functools.cache`
- Для распределённого кэша — Redis; ключи делай детерминированными и версионируемыми

## Retry и устойчивость
- Все обращения к внешним сервисам оборачивай в retry с экспоненциальным backoff
- Добавляй jitter, чтобы избежать thundering herd
- Задавай явные таймауты — никогда не оставляй вызов без `timeout=`
- Реализуй circuit breaker для зависимостей с нестабильным SLA

## Очереди и фоновые задачи
- Тяжёлые операции (отправка email, генерация отчётов) выноси в фоновые воркеры
- Используй идемпотентные задачи — воркер может выполнить задачу дважды
- Храни состояние задачи в персистентном хранилище, а не в памяти процесса

## Работа с данными
- Используй пагинацию / курсоры вместо `OFFSET` на больших таблицах
- Не делай N+1 запросов — агрегируй данные батчами
- Для write-heavy сценариев рассматривай буферизацию и batch insert

## Наблюдаемость
- Логируй входные параметры, время выполнения и ошибки для каждого внешнего вызова
- Используй структурированные логи (JSON) — они удобнее для парсинга в ELK/Loki
- Добавляй метрики (счётчики, гистограммы latency) через `prometheus_client` или аналог
